# 제약조건 수준 분석

## 배경

논문의 핵심 취지는 "다양한 configuration에 대한 데이터베이스 구축 및 추력 프로파일 유형 분석"이다. 제약조건을 어느 수준까지 고려할지에 따라 문제의 일반화 가능성과 실용성이 달라진다.

## 고려 사항

### 연구 목적
- 재사용 우주비행체의 신속한 임무궤도 설계를 위한 연구

### 임무 특성 예시
- Sub-satellite point가 특정 Geolocation(lon, lat)을 특정 시각에 지나가야 하는 임무
- 이 경우 terminal orbit의 특정 위치와 특정 시각이 함께 제약조건으로 주어짐

### 문제점
- 너무 specific한 제약조건은 일반화를 어렵게 함
- 데이터베이스 기반 분석이라는 논문 취지와 충돌
- 해를 찾기 어렵거나 문제를 풀기 어려워질 수 있음

---

## 제약조건 수준 옵션

| 수준 | 내용 | 장점 | 단점 |
|-----|------|------|------|
| **Level 1** (최소) | 출발/도착 **궤도**만 지정, 궤도상 위치 free, 전이시간 free | 가장 일반적, DB 구축 용이 | 실제 임무와 괴리 |
| **Level 2** (중간) | 궤도 지정 + **전이시간(duration)** 파라미터화 | 일반화 + 실용성 균형 | - |
| **Level 3** (구체적) | 특정 시각 + 특정 위치 통과 조건 | 실제 임무 반영 | 일반화 어려움, DB 구축 복잡 |

---

## 선택: Level 2 (중간 수준)

### 경계조건 정의

```
출발 궤도: (a₀, e₀, i₀, Ω₀, ω₀)  ← 궤도 요소로 지정
도착 궤도: (aₓ, eₓ, iₓ, Ωₓ, ωₓ)  ← 궤도 요소로 지정

출발 위치 (true anomaly): ν₀  ← free (최적화 변수)
도착 위치 (true anomaly): νₓ  ← free (최적화 변수)

전이 시간: T = tₓ - t₀  ← 파라미터로 설정 (DB 구축 시 변화)
```

### 이 접근의 장점

1. **일반화 가능**: 궤도 configuration만으로 정의되므로 다양한 케이스 생성 가능
2. **전이 시간의 영향 분석**: 전이 시간을 파라미터로 변화시키며 추력 프로파일 유형 변화 분석 가능
3. **물리적 의미 명확**: "주어진 전이 시간 내에 최적 궤적 찾기"
4. **Julian date 불필요**: 절대 시각 대신 상대 시간(duration)만 사용

### 수정 방향

**제거할 내용:**
- Julian date 관련 설명
- Time window 제약 (t₀, tₓ의 절대값 범위)

**추가할 내용:**
- 궤도 요소 기반 경계조건
- 전이 시간 T를 파라미터로 설정
- 출발/도착 true anomaly가 free임을 명시

## 향후 확장 가능성 (Discussion에서 언급)

> "본 연구에서는 일반적인 궤도 configuration에 대한 분석을 수행하였다. 실제 임무 적용 시에는 특정 시각에 특정 지상점(sub-satellite point)을 통과해야 하는 등의 추가 제약조건이 필요할 수 있으며, 이는 본 연구에서 도출된 추력 프로파일 유형 분류를 기반으로 초기 추정치를 설정한 후 정밀 최적화를 수행하는 방식으로 활용할 수 있다."

## 결론

데이터베이스 기반 분석이라는 논문 취지를 살리면서도 실용성을 확보하기 위해 **Level 2 (중간 수준)**를 채택한다:

- 궤도 요소로 출발/도착 궤도 정의
- 궤도상 위치(true anomaly)는 최적화 변수
- 전이 시간은 파라미터로 다양하게 설정
- 절대 시각(Julian date) 대신 상대 시간(duration) 사용
